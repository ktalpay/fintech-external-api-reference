# 0002: Store Only Token Hashes

## Status

Accepted

## Context

API keys are bearer credentials. If raw tokens are stored, database access, logs, support tooling, or backups could expose credentials that external clients can use directly.

The platform still needs to validate presented API keys and support operational review through token identifiers, prefixes, lifecycle status, and audit records.

## Decision

The platform stores only token hashes and non-secret token metadata. Raw API key tokens are generated by the platform, displayed once to the creator, and never stored or displayed again.

Lookup may use a non-secret key prefix or identifier to find candidate records before validating the presented token against the stored token hash.

## Consequences

- A lost raw token cannot be retrieved and must be replaced or rotated.
- Logs, audit events, and support views can reference key identifiers without exposing credentials.
- Database exposure does not directly reveal usable raw API key tokens.
- Implementations must choose appropriate token generation and hash verification mechanisms.
- Operational tooling must be designed around rotation rather than token recovery.

## Alternatives Considered

- Store raw tokens encrypted for later display: rejected because retrieval increases credential exposure risk and complicates access control.
- Store raw tokens in logs for troubleshooting: rejected because logs are not credential stores.
- Use only a key prefix as the credential: rejected because prefixes are not secret and are unsuitable for authentication.
